# Detector Plan

更新时间：2026-03-25

## 目标

把 `ai-writing-bad-cases-app` 做成一个“风格和风险审查器”：

- 检测 AIGC 常见坏味道
- 检测与作者画像不一致的地方
- 输出段落级证据和修订方向
- 作为博客写作 agent 的外部审查工具

本仓库不负责定义最终写作 prompt，也不负责直接产出最终博客正文。

## 产品定位

当前 README 已经定义了核心边界：

- 读取 `ai-writing-bad-cases/data/` 的结构化数据
- 做规则、统计、PPL、语义召回等检测
- 对高风险段落输出分析结果

下一步不是扩写研究口号，而是把这些能力统一成稳定的审查接口。

## 核心交付

### 1. 统一 JSON 输出

需要一个稳定 schema，供博客仓库的 review/revise agent 消费：

- 文档级总分
- 段落级风险
- 命中的信号列表
- 对应证据
- 建议动作
- 是否继续下一轮修订

优先级：最高

### 2. 信号分层

#### L1：规则和风格特征

- 套话和禁用词
- 模板化过渡
- 强迫性总结
- 对举句式
- 双引号滥用
- 句长均匀
- 对称句式堆叠
- 元话语过密

这层优先从 `.claude/skills/new-or-review/SKILL.md` 的“AI 痕迹检测”部分提取。

#### L2：作者风格偏差

- 开头是否落在具体问题或场景
- 是否有命令、数据、ASCII 图、表格
- 是否体现 V1 到 V2 到 V3 迭代
- 是否保留短句和闲笔
- 是否写出不确定性和边界
- 是否在不熟领域过度展开

这层的来源是博客仓库的 [`CLAUDE.md`](/home/yhh/learn/huanghuoguoguo.github.io/CLAUDE.md) 与已有文章风格。

#### L3：语义召回与概率特征

- seekdb 召回 bad-case 相似段
- 可选 PPL 检测
- 后续再评估更重的概率特征是否值得接入

这层暂时作为增强项，不抢前两层优先级。

## 数据建设

### bad-case 数据

需要新增两类数据：

1. 传统 AIGC 坏味道样本
2. “不像作者”的反例样本

建议按文体组织：

- 技术深潜
- 系统设计
- 读书笔记
- 实践复盘

每条 case 至少包含：

- `label`
- `pattern_text`
- `reason`
- `rewrite_direction`
- `article_type`
- `severity`

### 正样本参考

长期建议增加一个只读参考集，来自博客仓库中较满意的文章，用于 author-fit 校准。

## 里程碑

### Milestone 1：审查接口可用

- 输出稳定 JSON
- CLI 支持 `--format json`
- 给每个命中项稳定 `code`
- 给出 `stop_or_retry`

完成标准：

- 博客仓库可直接消费检测输出，生成 revision plan。

### Milestone 2：作者风格检测可用

- 把 `new-or-review` 中的 AI 痕迹清单结构化
- 新增 author-fit 规则集
- 输出 “为什么不像作者” 的具体理由

完成标准：

- 一篇文章至少能被指出 3 到 5 类明确风格偏差。

### Milestone 3：召回和概率增强

- seekdb 检索接入正式流程
- PPL 结果进入统一打分逻辑
- 评估误报率并调整阈值

完成标准：

- 对“流畅但空”的文章检出效果明显提升。

## 具体任务

1. 设计并冻结 JSON schema。
2. 给现有分析结果补 `code`、`reason`、`rewrite_hint`。
3. 从博客 skill 中提炼首批规则集。
4. 增加 author-fit 分数和说明。
5. 补一组金样例，验证输出稳定性。

## 不做的事

- 不直接生成整篇博客
- 不定义博客仓库里的 prompt 细节
- 不为了“更先进”先上重模型
- 不追求一个万能总分掩盖具体问题

## 验收标准

- 输出对 agent 友好，而不是只对人友好
- 能按段落给出证据，不只给总评
- 同一篇文章复跑，字段语义稳定
- 对博客修订有明确指导价值
